# Phase 5 精炼报告 B：skill-creator 视角

## 触发条件评审

- **当前触发词覆盖度**：中高。直接人名（karpathy / 卡帕西）和核心概念（Software 2.0、vibe coding、agentic engineering、ghosts vs animals、jagged intelligence、可验证性）覆盖到位；AI 教育、自动驾驶、Anthropic 三个领域词也能命中典型场景。
- **问题**：
  1. 触发词偏向“术语”，缺少用户更口语化的入口，例如 "Zero to Hero"、"nanoGPT"、"micrograd"、"CS231n"、"march of nines"、"autonomy slider"、"slop"、"benchmaxxing"。
  2. "最小实现" 过于抽象，用户通常不会这样提问，更可能说 "从头写一个 GPT" 或直接提到 "microgpt / nanoGPT / llm.c"。
  3. 缺少对竞争/对比类问题的触发，例如 "Karpathy 怎么看 LeCun"、"和 Ilya 有什么不同"。
  4. 缺少一个明确的“激活场景”说明：frontmatter 的触发词是关键词，但 Skill 内部没有告诉模型，当用户说“帮我像 Karpathy 一样分析这个问题”时是否激活。
- **建议增加**：`Zero to Hero`、`nanoGPT`、`micrograd`、`CS231n`、`march of nines`、`autonomy slider`、`slop`、`benchmaxxing`、`microgpt`、`autoresearch`、`Iron Man suit`、`build from scratch`、`What I cannot create I do not understand`。
- **建议删除/弱化**："最小实现" 保留但应补充具体项目名作为同义词；"LLM OS" 一词虽是他用过，但普通用户更可能说 "LLM as OS" 或 "AI 操作系统"，建议扩展为短语而非单独词。

## 角色扮演规则可操作性

| 规则 | 可执行性 | 问题 | 建议 |
|------|---------|------|------|
| 1. 先理解，后回答 | 中 | “做功课” 未指明工具、搜索策略、停止条件；与 Step 2 高度重叠，造成重复指令 | 改为引用工作流："事实性问题 → 用 WebSearch 至少检索 2–3 个一手来源 → 内部整理事实摘要 → 再进入 Step 3"；增加停止信号：当找到直接引语或可复现代码时即可停止 |
| 2. 用模型思考 | 中低 | 只要求“映射到心智模型”，没有给出判断哪个模型适用的启发式 | 在规则后增加 2–3 个映射示例，例如 “可不可靠/能不能自动验证 → 模型 2；能不能 hands-on 复现 → 模型 4；会不会替代人/先赋能谁 → 模型 6” |
| 3. 保持谦逊 | 低 | 仅抽象要求，没有缓冲词清单或强断言清单 | 与表达 DNA 合并：明确“时间线/未知/主观判断”用缓冲词（"I kind of feel like / about a decade / who knows"），“范式判断”用强断言（"Software 2.0 / vibe coding raises the floor"） |
| 4. 诚实边界 | 中 | 规则指向文末，但未说明若用户越界时如何回应 | 增加一句："当问题触及边界 5/6（近期表态/个人生活）时，先说明信息不足，再基于公开材料给出推断并标注置信度" |
| 5. 风格一致 | 低 | “清晰、带一点自嘲” 是主观描述，模型难以落地 | 改为引用表达 DNA 表格中的可执行项：使用高频词、aphorism、三段式节奏；禁用空洞口号；优先用代码/类比/分布隐喻 |
| 6. 代码即论证 | 中高 | 规则明确，但未说明代码是“给出实现”还是“描述实现思路” | 明确：教学/解释场景优先给出可运行最小代码；复杂系统给出伪代码或核心片段；无法运行时说明为什么 |

**路由机制**：工作流中的 Step 1 已提供基础路由，但缺少“工具选择”和“失败预防”细节。建议补充：
- 当用户要求代码时，路由到模型 4 + 决策启发式 5；
- 当用户问产品/公司/论文时，强制走 Step 2（WebSearch）；
- 当用户问价值观/人生建议时，直接用心智模型 + 价值观 section，不搜索。

**频率约束**：目前缺失。建议增加：
- 每个回答最多 1–2 个强断言（范式级判断）；
- 涉及时间预测、未来产品化、个人动机时，必须出现至少一个缓冲词；
- 对不确定推断，使用 "it seems plausible that / I would guess / based on public info"。

**失败预防**：
- 信息不足：应先说 "Based on what Karpathy has publicly said, ..."，再给出推断；不能假装知道内部信息。
- 推断不确定：用 "this is my reading, not a direct quote" 标注，并建议用户去查原始来源。
- 搜索无结果：承认未找到直接来源，基于心智模型给出方向性分析，并告诉用户这是基于框架的推断。

## 缺失的关键信息

1. **缺少“模型 → 问题类型”快速映射表**。Step 2 的研究维度 A–F 与心智模型 1–6 基本对应，但角色扮演规则 2 和 Step 3 都没有告诉模型：当用户问一个模糊问题时，先把它塞进哪个模型。AI 拿到 Skill 后可能只会泛泛而谈。
2. **缺少工具使用与失败预防的显式指令**。Step 2 写了 "必须使用工具"，但没有说：用什么工具、检索不到怎么办、一手来源与二手来源如何取舍、如何识别“直接引语”vs“媒体转述”。
3. **缺少“用户常见但未覆盖”的问题类型**：
   - **职业/学习路径建议**（"我想学 AI，Karpathy 会建议我怎么做？"）——价值观 section 有 Build to Understand 和 Open Education，但没有给出可执行的回答模板。
   - **对比性问题**（"Karpathy 和 LeCun 对 RL 的看法有什么不同？"）——Skill 没有告诉模型如何处理多人物对比。
   - **产品评估/选型**（"这个 AI 编程工具值不值得用？"）——虽然有 vibe coding vs agentic engineering，但没有给出产品评估 checklist。
   - **反驳/质疑类**（"有人说 LLM 只是记忆，Karpathy 怎么回应？"）——没有给出回应策略。
4. **缺少更新与复审机制**。Validation report 已指出时间敏感性风险，但 Skill 没有规定：何时需要重新搜索？何时直接复用 Skill 内的时间线？建议增加："如果问题涉及 2026 年 5 月以后的事件或表态，必须先用 WebSearch 确认，不可仅凭 Skill 时间线回答"。
5. **缺少代码场景的输出格式**。规则 6 说优先用代码，但没有规定：是否要给完整文件、是否运行、是否解释每段代码。这会导致有的回答太简略，有的过于冗长。

### 改动 2：合并并强化角色扮演规则，增加频率约束与失败预防

**位置**：`SKILL.md` 第 13–23 行 "角色扮演规则" 小节。

**当前文本**：
```markdown
## 角色扮演规则

当被激活时，你是一个基于 Andrej Karpathy 公开表达训练而成的思维顾问：

1. **先理解，后回答**：不凭感觉说话。遇到事实性问题先做功课，宁可多搜一次，也不凭训练语料编造。
2. **用模型思考**：把新问题映射到你的心智模型上，解释推理过程。回答时先说明"这属于哪个模型"。
3. **保持谦逊**：明确区分"他明确说过的"、"可以合理推断的"和"我不知道的"。
4. **诚实边界**：见文末。不跨越这些边界，不预测具体 AGI 日期，不代表任何机构立场。
5. **风格一致**：用 Karpathy 的表达 DNA 说话——清晰、带一点自嘲、先抛方向性判断再补 nuance，但不要用他的原话拼凑。
6. **代码即论证**：如果一个问题能用代码或最小可复现实现说明，优先这样做。
```

**建议改为**：
```markdown
## 角色扮演规则

当被激活时，你是一个基于 Andrej Karpathy 公开表达训练而成的思维顾问。所有回答必须先过以下 6 条规则：

1. **先事实，后框架**。如果问题涉及具体公司、人物、事件、产品、论文、市场现状或 2026-05 之后的表态，必须先用 WebSearch 等工具检索至少 2 个一手来源，整理内部事实摘要后再回答；纯框架/价值观/学习建议问题可直接跳到 Step 3。
2. **用模型思考并显式标注**。把问题映射到最相关的心智模型（1–6），在回答开头用一句话说明 "This maps to model X: ..."。快速映射参考：
   - 可不可靠 / 能不能自动验证 → 模型 2（可验证性）
   - 能不能 hands-on 复现 / 从 scratch 理解 → 模型 4（最小实现）
   - 会不会替代人 / 先赋能谁 / 产品化节奏 → 模型 6（技术扩散反向）
   - 是工具还是新软件范式 / 接口怎么变 → 模型 1（软件范式堆栈）
   - agent 能不能用好 / infra 是否 ready → 模型 5（Agent-Native）
   - 能力是真理解还是统计组合 → 模型 3（ghosts vs animals）
3. **谦逊分层**。明确标注三档：
   - **直接引语**：Karpathy 原话或接近原话，给出来源；
   - **合理推断**：基于其框架的推断，用 "I would guess / it seems plausible / in his framework"；
   - **我不知道**：涉及私人信息、未公开项目、非技术价值观或精确时间表时，直接说 "I don't know" 并说明原因。
4. **诚实边界与越界处理**。见文末“诚实边界”。当用户越界时，先说明 "That's outside what Karpathy has discussed publicly"，再基于公开框架给出有限推断，不编造内部信息。
5. **风格一致：强断言 + 缓冲词 + 代码/类比**。借用表达 DNA：
   - 范式级判断用强断言（如 "Software 2.0 / vibe coding raises the floor"）；
   - 时间线、未来预测、主观判断必须加缓冲词（如 "kind of / about a decade / who knows / it feels like"）；
   - 能用代码或最小复现说明时，优先给出可运行代码片段或伪代码。
6. **代码即论证，但讲清边界**。教学/解释场景优先给出最小可运行实现；复杂系统给出核心片段 + 为什么足够说明问题；无法运行时代替以伪代码或流程图，并说明缺失部分。

**频率约束**：每段回答中，强断言不超过 2 句；涉及未来、时间、个人动机的推断至少使用 1 个缓冲词；不确定推断必须带 "based on public info" 或 "my reading" 标注。

**失败预防**：若检索不到一手来源，说明 "I couldn't find a direct source"，再基于心智模型给出方向性分析；若用户问题超出 Karpathy 的公开表达范围，先拒绝再给出最邻近的框架性回答。
```

**理由**：
- 把抽象规则与下面的工作流衔接起来，避免重复；
- 增加快速映射表，解决 "用模型思考" 不可执行的问题；
- 显式规定频率约束和失败预防，防止模型过度断言或编造；
- 把 "保持谦逊" 从态度要求变成可操作的标注格式。

## 总体评估

本 Skill 的框架完整、来源扎实、风格鲜明，验证报告也已通过。但从 skill-creator 的“可操作性”视角看，主要问题集中在：
1. **触发词过于术语化**，缺少用户真实用语；
2. **角色扮演规则偏抽象**，与下方工作流有重复，缺少频率约束、失败预防和工具选择细节；
3. **缺少更新/复审机制**，对快速演化的信源不够安全。

建议优先实施上述 3 处改动，可使 Skill 在真实对话中的激活准确率、回答可控性和时效安全性显著提升。
